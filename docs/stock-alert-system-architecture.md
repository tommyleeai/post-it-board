# 股票監控系統與 AI 判定架構總結

本文檔記錄了白板系統中「股票警報與監控功能」的架構設計、AI 語意判斷機制，以及相關的 UI/UX 與動畫優化決策。供後續維護與功能擴充參考。

---

## 1. 系統架構總覽 (Architecture Overview)

股票監控系統高度整合了多個子模組，其核心哲學為 **「資料綁定於卡片，邏輯集中於服務」**：

*   **資料層 (Yjs CRDT)**：所有的警報設定（包含 `symbol`、`targetPrice`、`condition`、`status` 及通知選項 `options`）皆直接作為屬性，附加在該張股票卡片的 `PostIt.Note.getCache()` 資料結構中。這確保了不同裝置與分頁間的即時同步。
*   **介面層 (`board_v2.js` & `stock_alert.js`)**：
    *   `board_v2.js` 負責處理卡片點擊、Focus Mode 燈箱動畫、翻轉等 DOM 操作。
    *   `stock_alert.js` 提供背景輪詢 (Polling) 機制，定期掃描所有狀態為 `watching` 的卡片，並呼叫外部 API 與 AI 服務進行驗證。同時負責渲染「股票監控面板 (Dashboard)」。
*   **邏輯層 (`ai.js` & Node.js API)**：負責解析使用者的自然語言設定，並將真實市場數據與之比對。

---

## 2. AI 語意判斷引擎 (AI Condition Evaluation)

傳統的股價警報通常強制使用者選擇「大於/小於」加上具體數字。為了提供更流暢的白板體驗，我們導入了 AI 語意解析：

### 2.1 判斷流程
1.  使用者在卡片背面輸入自然語言條件（例如：「漲破 400 通知我」、「跌破均線」、「大跌超過 5%」）。
2.  `stock_alert.js` 在輪詢時，會先透過 Node.js 的 `/api/stock` 抓取最即時的真實市場報價 (`currentPrice`)。
3.  系統將 `[設定條件]`、`[目標股價]` 與 `[真實當前股價]` 打包，發送至 `ai.js` 中的 `evaluateStockCondition` 方法。
4.  利用 Google Gemini API 進行語意分析，回傳嚴格的 JSON 格式 `{ "triggered": true/false }`。

> [!TIP]
> **無頭驗證優勢**：驗證股價時，全面採用 Node.js 的無頭請求 (`fetch`)，放棄容易因跨域或權限問題失敗的瀏覽器端爬蟲，確保了判斷的 100% 可靠性與低延遲（毫秒級回覆）。

## 3. UI/UX 與動畫設計決策 (Design & Animation)

在實作卡片互動時，最大的挑戰來自於 **CSS 動畫與 Yjs 即時重繪的衝突 (Race Condition)**。

### 3.1 Focus Mode 與翻轉動畫防護
當使用者按下「設定警報並儲存」時，卡片需要完成兩個動作：從背面翻轉回正面（300ms），並從畫面中央飛回原始位置（500ms）。
**問題**：如果立即將狀態寫入 Yjs，其他客戶端或本地的 `onSnapshot` 會瞬間重繪該張卡片，導致 DOM 被抽換，原本順暢飛行的 CSS 動畫會立刻中斷，卡片「瞬間瞬移」回原處。
**解決方案**：在 `board_v2.js` 的儲存與刪除邏輯中，我們加入了 **850ms 的延遲寫入 (setTimeout)**。確保卡片完美落地後，才更新資料庫，大幅提升了流暢的高級 UX 體驗。

### 3.2 股票監控儀表板 (Dashboard)
為了避免使用者忘記到底在哪幾張卡片上設定了警報，我們開發了全局的監控面板：
*   **動態掃描**：面板不需要獨立的資料庫 Collection，它每次開啟時會直接掃描 `PostIt.Note` 快取，過濾出 `status: 'watching'` 的卡片。
*   **通知狀態可視化**：在面板清單中，直接以圖示 (Icon) 顯示該警報啟用的通知方式（文字 Toast、警報音效 Sound、語音播報 TTS），讓設定一目了然。
*   **一鍵定位 (Locate)**：點擊面板上的定位按鈕，系統會使用 `bringToFront` 將目標卡片拉至最高層次 (`Z-index`)，並加上為期 3 秒的 `highlight-glow` CSS 發光特效，讓使用者在一片混亂的白板中也能一眼找到。
*   **快速移除 (Remove)**：使用者無須將視角移動回原始卡片，可直接在面板內點擊「移除」，系統會立刻同步 Yjs 解除監控，並即時重繪面板與白板卡片狀態。

## 4. 瀑布流網格排列 (Masonry Auto-Sort)

為了修復舊版「自動排列」會導致長卡片重疊、短卡片下方留白的問題，我們全面升級了演算法：

*   **動態高度計算**：放棄寫死的 300px 高度。透過讀取每張卡片真實的 DOM `offsetHeight`，確保間距吻合。
*   **Masonry 邏輯**：建立一個陣列追蹤每一行 (Column) 的當前總高度。每次放置新卡片時，永遠尋找「**目前高度最短的那一行**」進行安插。
*   **群組保護 (Group Protection)**：自動排列前，強制過濾掉帶有 `groupId` 的子卡片，只對「獨立卡片」與「群組首頁卡片」重新排版，避免破壞使用者辛苦建立的群組結構。
*   **智能置中**：根據當前解析度與總列數計算 `offsetX`，讓整個瀑布流在寬螢幕上能完美置中。

## 5. 實時載入進度條 (Global Progress Loader)

隨著系統導入 Yjs 模組與千筆等級的卡片資料，白板初次載入的白畫面時間增加。
*   **極早期注入**：在 `index.html` 的 `<head>` 中便宣告 `updateLoader` API 與遮罩 CSS，確保在任何笨重的 JS 載入前就開始運作。
*   **精準埋點追蹤**：進度條綁定了 7 大真實系統啟動階段（核心腳本 10%、登入驗證 30%、讀取設定 50%、連接協作引擎 70%、載入快取 80%、與雲端同步 90%、完成繪製 100%）。
*   讓使用者明確知道系統並未當機，提升容忍度與信賴感。
